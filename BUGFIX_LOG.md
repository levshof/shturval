# BUGFIX_LOG — WB Shturval

Журнал значимых багов. Цель — чинить **настоящую причину**, а не симптом, и не
превращать продукт в набор скрытых обходов (спец-раздел 0.4).

## Правила записи

Каждый нетривиальный багфикс фиксируем по шаблону. Запрещено:
- молча возвращать пусто / подменять «странные» данные без явного статуса ошибки;
- дублировать формулу, чтобы «там посчиталось иначе» (один источник правды);
- временный обход без пометки в этом журнале и в `DECISIONS.md` (тех-долг).

## Шаблон записи

```
### BUG-XXXX — Короткий заголовок
- **Дата:** YYYY-MM-DD
- **Симптом:** что наблюдалось (со стороны пользователя/логов).
- **Настоящая причина:** корневая причина, а не симптом.
- **Затронутые модули:** файлы/слои.
- **Что изменено:** суть фикса.
- **Как проверено:** тест / QA-сценарий / валидация / диагностический лог.
- **Почему это не костыль:** почему решение устраняет причину.
- **Остаточный риск:** есть/нет, какой.
- **Будущая оптимизация:** нужна/нет.
```

## Записи

### BUG-0001 — Расходы на рекламу «по нулям» у всех товаров
- **Дата:** 2026-07-01
- **Симптом:** после синхронизации реклама = 0 ₽ у всех товаров; в карточке баннер «Нет данных по рекламе».
- **Настоящая причина:** `adv/v3/fullstats` отвечает `400` на **весь** батч кампаний, если хотя бы у одной из них нет статистики за период (архивные / ни разу не показанные кампании). `fetchAdvFullStats` тянул все id из `promotion/count` (включая завершённые) пачками по 50; один «мёртвый» id ронял весь запрос, ошибка глоталась в `try/catch` шага рекламы (`warn`), и `ad_stats` оставался пустым.
- **Затронутые модули:** `backend/src/wb/client.ts` (`fetchAdvFullStats`, новый `advFullStatsChunk`).
- **Что изменено:** при `400` на пачку — переспрашиваем кампании по одной и пропускаем только те, что реально пусты; остальные расходы сохраняются.
- **Как проверено:** typecheck + существующие тесты зелёные; логика подтверждена по официальной доке WB и обсуждению на форуме (`dev.wildberries.ru/forum/1441`) о `400` для кампаний без показов.
- **Почему это не костыль:** устраняет корневую причину (нетолерантность к «пустым» кампаниям), а не прячет её; данные не подменяются, пропуск логируется.
- **Остаточный риск:** при `400` пачки делаем до 50 доп. запросов (интервал 1.5с) — медленнее, но только в проблемном случае.
- **Будущая оптимизация:** можно заранее отфильтровывать кампании по статусу из `promotion/count`, чтобы не переспрашивать поштучно.

### BUG-0002 — Прибыль/маржа/средняя цена считались по разным периодам
- **Дата:** 2026-07-01
- **Симптом:** при наличии финотчёта прибыль и маржа выглядели заниженными, средняя цена — «неправильной».
- **Настоящая причина:** выручка и выплата брались из **закрытых недель** финотчёта, а себестоимость/производные — из **всех продаж за 30 дней** (`units30`), куда входят ещё не отражённые в отчёте продажи. Разные периоды → `cost = unitCost × units30` завышен, `profit`/`marginPercent`/`avgPrice` искажены.
- **Затронутые модули:** `backend/src/services/economics.ts`, `backend/src/sync/recompute.ts`, `backend/src/services/products.ts`.
- **Что изменено:** `summarizeFinanceWindow` теперь считает и `financeUnits` (нетто по отчёту, возвраты вычитаются); при наличии финотчёта экономика (себестоимость, прибыль, средняя цена, прибыль/шт) считается по этим же «проверенным» единицам. Без финотчёта — как раньше, по продажам за 30 дней.
- **Как проверено:** новый `backend/src/services/economics.test.ts` (4 теста) + полный прогон (84 теста) зелёный.
- **Почему это не костыль:** приводит выручку и количество к одному периоду — соответствует принципу «проверенные недели» (D-0009), формула остаётся единой (`computeEconomics`).
- **Остаточный риск:** в карточке «Продано» при наличии отчёта показывает отражённые единицы (может быть меньше фактических продаж за 30 дней) — это честнее, но отличается от скорости продаж в KPI.
- **Будущая оптимизация:** явно подписать в UI, что экономика — по проверенным неделям, а скорость — по 30 дням.

### BUG-0003 — Себестоимость завышена в 3-4 раза (единицы считались по ВСЕМ строкам финотчёта)

- **Дата:** 2026-07-02
- **Симптом:** в карточке товара себестоимость (`102 530 ₽`) не билась с `продано × себестоимость/шт` (`264 × 91.3 ≈ 24 100 ₽`); из-за этого «Средняя цена» занижена (94 ₽ вместо ~400 ₽), прибыль и маржа уходили в минус на прибыльном товаре.
- **Настоящая причина:** `summarizeFinanceWindow` (`backend/src/services/economics.ts`) суммировала `quantity` по **всем** строкам финотчёта WB `reportDetailByPeriod` в окне, а не только по строкам продаж/возвратов. Отчёт содержит отдельные строки по операциям — логистика, хранение, штрафы, платная приёмка, удержания — и они тоже несут `quantity`. Сумма квант по всем операциям завышала количество единиц (~1127 вместо 264 фактически проданных), и `cost = unitCost × units` считался по раздутому числу.
- **Затронутые модули:** `backend/src/services/economics.ts` (`summarizeFinanceWindow`), `backend/src/services/products.ts` (карточка товара).
- **Что изменено:** строка засчитывается в `units`/`revenue` только когда это реальная строка продажи/возврата (индикатор — ненулевой `retail_amount`, который WB заполняет только на продажах/возвратах). Выплата (`ppvz_for_pay`) и расходные части считаются как прежде — по всем строкам. «Продано» в карточке теперь берётся из того же источника (`view.economics.units`), что и себестоимость/средняя цена/прибыль-на-штуку, — все производные снова на одном периоде.
- **Как проверено:** новый тест в `backend/src/services/economics.test.ts` («ignores non-sale operation rows…») с явными строками логистики/хранения/приёмки, несущими `quantity` без `retail_amount`; полный прогон (90 тестов) зелёный.
- **Почему это не костыль:** устраняет причину (смешение типов операций при подсчёте единиц), а не подгоняет число под ожидаемое; выручка не менялась (она и раньше считалась только по продажам/возвратам — `retail_amount` на прочих строках отсутствует), поэтому исправление точечное.
- **Остаточный риск:** нет — используемый признак (`retail_amount != 0`) надёжно отличает продажи/возвраты от прочих операций в текущем формате отчёта WB.
- **Будущая оптимизация:** при появлении в отчёте поля типа операции (`supplier_oper_name`) можно заменить эвристику на явную фильтрацию по типу — не требуется сейчас.

### BUG-0004 — Реклама «по нулям» у большинства товаров (расход не разбит WB по nmId)

- **Дата:** 2026-07-02
- **Симптом:** после успешной синхронизации у большинства товаров «Реклама» = 0 ₽, хотя продавец точно тратил деньги на их продвижение (в т.ч. видно в кабинете WB).
- **Настоящая причина:** `persistAds` (`backend/src/sync/syncEngine.ts`) брал расход **только** из вложенной разбивки по товарам `days[].apps[].nm[].sum`. По подтверждённому на форуме WB API поведению (`dev.wildberries.ru/forum/1441`, уже цитировался в BUG-0001) `adv/v3/fullstats` может отдавать реальный дневной итог по кампании, но нулевую или неполную разбивку по `nm[]` — особенно на автокампаниях. Расход, не попавший в `nm[]`, тихо терялся: товар оставался без строки `ad_stats` и показывал 0 ₽.
- **Затронутые модули:** `backend/src/wb/types.ts` (день. итог `sum`/`views`/`clicks` в `WbAdvFullStat.days[]`), новый `backend/src/domain/ads.ts` (`allocateAdSpend`), `backend/src/sync/syncEngine.ts` (`persistAds`), `backend/src/services/economics.ts` / `backend/src/sync/recompute.ts` / `backend/src/services/products.ts` (флаг `adEstimated`).
- **Что изменено:** дневной итог WB (`days[].sum`) теперь берётся как источник правды «сколько потрачено»; точная разбивка по `nm[]` остаётся как есть (`source = PRECISE`). Неразнесённый остаток (итог минус известная разбивка) распределяется пропорционально между товарами той же кампании по их доле точного расхода за всё окно (`source = ALLOCATED`); если у кампании вообще нет ни одной строки с товарами за весь период — остаток не выдумывается, а попадает в диагностику шага синка «реклама» (сколько ₽ не удалось привязать). Экономика товара помечает `adEstimated = true`, если в его расходе участвовала распределённая часть.
- **Как проверено:** новый `backend/src/domain/ads.test.ts` (5 тестов: точная разбивка проходит без изменений; пропорциональное распределение остатка по историческому соотношению трат; равное распределение, когда ни у одного товара кампании нет точного расхода; остаток без единого известного товара уходит в `unattributed`, а не теряется молча; слияние повторных `nm`-записей в одном дне). Полный прогон (90 тестов) и typecheck (backend+frontend) зелёные.
- **Почему это не костыль:** не подменяет отсутствующие данные придуманными числами — либо честно распределяет по уже известной пропорции трат того же товара/кампании (это его собственные деньги, а не чужие), либо явно помечает как непривязанные (спец-раздел 0.4). Источник (`AdSource.PRECISE`/`ALLOCATED`) в схеме БД для этого и предназначен.
- **Остаточный риск:** день-уровневое поле `days[].sum` в `adv/v3/fullstats` не подтверждено вживую (официальная документация WB отдаёт контент только после JS-проверки, недоступно для автоматической сверки — см. `WILDBERRIES_API_NOTES.md` §0/§12). Код рассчитан на мягкую деградацию: если поля нет, `totalSpend` берётся равным уже известной точной сумме — тогда распределение остатка просто не происходит (поведение как до фикса), без риска исказить данные. Проверить на живом ключе при следующей синхронизации и свериться с суммой в кабинете WB.
- **Будущая оптимизация:** если день-уровневый `sum` подтвердится — задокументировать в `WILDBERRIES_API_NOTES.md` §8 как проверенный (`✅ verified`).

- **Дата:** 2026-07-01
- **Симптом:** «трудно понять, почему прибыль/расходы/реклама не считаются».
- **Причина:** бэкенд писал статусы шагов синка (`steps`, в т.ч. `warn`: «нет доступа к разделу реклама», «финансовый отчёт …»), но фронт их **нигде не показывал**.
- **Затронутые модули:** `frontend/src/pages/DashboardPage.tsx` (новый блок `SyncDiagnostics`).

- **Почему это не костыль:** делает уже собираемую диагностику видимой — это и есть честный статус источников (спец-раздел 0.4), а не сокрытие проблемы.

### BUG-0005 — Реклама «по нулям», а таблица `ad_stats` пустая — без единого сигнала почему
- **Дата:** 2026-07-02
- **Симптом:** в карточках всех товаров «Реклама» = 0 ₽, в таблице `ad_stats` **ноль строк** после множества синхронизаций, хотя у продавца точно есть активные рекламные кампании. Никаких предупреждений на дашборде.
- **Настоящая причина:** шаг синка «реклама» (`syncEngine.ts`) помечался `status: 'ok'` даже когда `persistAds` записал **ноль строк**. Кампании находились (`fetchAdvCampaignIds` вернул id), но `adv/v3/fullstats` за период отдавал пустой ответ либо нулевой расход (документированный сбой v3 с нулевыми `sum/views/clicks`, либо трат за 30 дней не было) — и это тихо превращалось в «ok, count 0» без баннера. Классический анти-паттерн «молча вернуть пусто» (спец-раздел 0.4): пустая `ad_stats` была следствием, а не причиной, и без диагностики причину нельзя было отличить (0 кампаний? нет доступа? нулевой ответ WB? все кампании отсеяны по 400?).
- **Затронутые модули:** `backend/src/sync/syncEngine.ts` (шаг 5 + `persistAds` теперь возвращает `totalSpendSeen`), `backend/src/wb/client.ts` (логи `promotion/count` и `fullstats`).
- **Что изменено:** если кампании найдены, но `persistAds` дал 0 строк — шаг помечается `warn` с конкретной причиной, различая «WB не вернул статистику ни по одной кампании» и «WB вернул статистику по N кампаниям, но с нулевым расходом» (по `adStats.length` и `totalSpendSeen`). Добавлены серверные логи по всему пути: сколько кампаний из `promotion/count`, сколько записей из `fullstats`, сколько строк записано, сколько ₽ увидено/не привязано.
- **Как проверено:** typecheck (backend) + полный прогон (90 тестов) зелёные. Диагностика проявится на живом ключе при следующей синхронизации — по баннеру/логам станет видно, какая именно стадия отдаёт пусто.
- **Почему это не костыль:** не подменяет данные и не прячет проблему — наоборот, делает невидимый ранее провал честно видимым (спец-раздел 0.4, продолжение DIAG-0001). Пустая `ad_stats` чинится не ручной правкой БД (она перезапишется синком), а устранением причины на стадии загрузки WB — которую теперь видно.
- **Остаточный риск:** сам контракт `adv/v3/fullstats` (GET `ids/beginDate/endDate`, поле `days[].sum`) остаётся `check-live` (`WILDBERRIES_API_NOTES.md` §8/§12) — если WB отдаёт 0 из-за неверного контракта, это подтвердится логами (`records > 0`, `totalSpendSeen = 0`) и потребует сверки с живой докой отдельным фиксом.
- **Будущая оптимизация:** когда по логам подтвердится реальный режим отказа WB — либо задокументировать `days[].sum` как `✅ verified`, либо поправить контракт v3 по живой доке.
