# SETUP_GUIDE.md — установка и запуск WB Shturval

Пошаговая инструкция «с нуля». Рассчитана на то, что вы открываете её и по шагам
всё настраиваете без догадок. Есть **три способа** запуска:

- **Способ A — Docker (рекомендуется для своего сервера/VPS).** Одна команда
  поднимает базу, backend и сайт.
- **Способ B — вручную.** Для разработки и тонкой настройки.
- **Способ C — бесплатный облачный хостинг (Supabase + Render + Vercel).** Без
  своего сервера, на бесплатных тарифах. Полная инструкция — в
  [§20.11](#2011-бесплатный-облачный-деплой-supabase--render--vercel).

> Главный принцип проекта: надёжность и отсутствие костылей. Если что-то идёт не
> так — смотрите раздел [20.8 Быстрая диагностика](#208-быстрая-диагностика).

---

## ⚡ Свой сервер — быстрый старт (Docker)

Если у вас свой сервер/VPS, этого блока достаточно. Нужен только установленный
**Docker** (с Docker Compose). База, backend и сайт поднимаются одной командой.

```bash
git clone <ваш-репозиторий> wb-shturval && cd wb-shturval

# Создать секреты
cp .env.example .env
node -e "console.log('JWT_SECRET='+require('crypto').randomBytes(48).toString('base64url'))"
node -e "console.log('ENCRYPTION_KEY='+require('crypto').randomBytes(32).toString('base64'))"
# Впишите эти два значения в .env, задайте POSTGRES_PASSWORD,
# а CORS_ORIGIN = адрес, по которому откроете сайт (напр. http://SERVER_IP:8080).

docker compose up -d --build      # собрать и запустить
```

Откройте **http://SERVER_IP:8080** → регистрация → Настройки → WB-ключ →
«Синхронизировать». Готово.

Дальше по необходимости:

| Действие | Команда |
|----------|---------|
| Логи | `docker compose logs -f backend` |
| Обновить после изменений | `git pull && docker compose up -d --build` |
| Остановить | `docker compose down` (данные сохранятся в volume) |
| HTTPS на домене | поставьте перед `web` обратный прокси (Caddy/Traefik/nginx) и выдайте сертификат; затем в `.env` укажите `COOKIE_SECURE=true` и `CORS_ORIGIN=https://ваш-домен`, и `docker compose up -d` |

Если что-то не так — [§20.8 Быстрая диагностика](#208-быстрая-диагностика).
Остальные разделы ниже (20.1–20.11) нужны только для ручной настройки, облака и
деталей — **для своего сервера их можно не читать**.

## 20.1. Что нужно заранее

| Что | Зачем | Где взять |
|-----|-------|-----------|
| **PostgreSQL 14+** | хранение данных | Docker (входит в compose), либо managed: Supabase / Neon / Railway / свой сервер |
| **Node.js 20+ и npm** | запуск backend и сборка frontend | https://nodejs.org (нужно только для способа B; в Docker уже внутри) |
| **Docker + Docker Compose** | способ A | https://docs.docker.com/get-docker/ |
| **API-ключ Wildberries** | синхронизация | кабинет продавца WB → Настройки → Доступ к API (см. [20.6](#206-настройка-синхронизации-wildberries)) |
| **Хостинг** (для боевого деплоя) | разместить сайт и backend | любой VPS с Docker, либо раздельно: Vercel/Netlify (frontend) + Railway/Render/Fly (backend) + Supabase/Neon (БД) |
| **Домен** (необязательно) | красивый адрес | любой регистратор |

Минимум для локального запуска: **только Docker**. Всё остальное он поднимет сам.

## 20.2. Переменные окружения и secret keys

Секреты **никогда не хранятся в коде**. Все ключи задаются через переменные
окружения. Ниже — полный список.

### Для docker-compose (файл `.env` рядом с `docker-compose.yml`)

Скопируйте `.env.example` → `.env` и заполните:

| Переменная | Что это | Где взять / как сгенерировать | Пример формата |
|-----------|---------|-------------------------------|----------------|
| `POSTGRES_USER` | пользователь БД | придумайте | `wb` |
| `POSTGRES_PASSWORD` | пароль БД | придумайте надёжный | `S7r0ng-Pass!` |
| `POSTGRES_DB` | имя БД | придумайте | `wb_shturval` |
| `CORS_ORIGIN` | адрес сайта (браузерный URL) | ваш домен или `http://localhost:8080` | `https://shturval.example.ru` |
| `JWT_SECRET` | подпись сессий входа | `node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"` | длинная случайная строка |
| `ENCRYPTION_KEY` | шифрование WB-ключа (32 байта base64) | `node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"` | `q1w2...==` (ровно 32 байта в base64) |
| `COOKIE_SECURE` | `true` только под HTTPS | — | `false` локально, `true` на проде |

### Для backend при ручном запуске (файл `backend/.env`)

Скопируйте `backend/.env.example` → `backend/.env`. Дополнительно к секретам выше там есть:

| Переменная | Что это | Пример |
|-----------|---------|--------|
| `DATABASE_URL` | строка подключения к Postgres | `postgresql://wb:wb@localhost:5432/wb_shturval?schema=public` |
| `PORT` | порт backend | `4000` |
| `WB_MIN_INTERVAL_MS` | пауза между запросами к одному методу WB (лимит ~1/мин) | `60000` |
| `WB_MAX_PAGES` | предохранитель от бесконечной пагинации | `200` |
| `WB_HISTORY_DAYS` | глубина первой выгрузки (WB хранит ~90 дней) | `90` |

### Для frontend (файл `frontend/.env`)

| Переменная | Что это | Пример |
|-----------|---------|--------|
| `VITE_API_URL` | адрес backend. **Пусто** в dev (прокси) и в Docker (nginx). На раздельном проде — URL backend | `https://api.shturval.example.ru` |

> ⚠️ Если потеряете `ENCRYPTION_KEY` — сохранённые WB-ключи нельзя будет
> расшифровать, их придётся ввести заново. Храните его в секретах хостинга.

## 20.3. Настройка базы данных

### Способ A (Docker) — ничего делать не нужно

`docker compose up` сам поднимет Postgres и применит миграции при старте backend
(команда `prisma migrate deploy` встроена в контейнер). Данные хранятся в volume
`pgdata` и переживают перезапуск.

### Способ B (вручную)

1. Создайте пустую базу (локально или в Supabase/Neon).
2. Пропишите `DATABASE_URL` в `backend/.env`.
3. Примените миграции:
   ```bash
   cd backend
   npm install
   npm run db:migrate     # prisma migrate deploy — применяет готовые миграции
   ```
4. Проверьте, что таблицы создались:
   ```bash
   npm run db:studio      # откроет Prisma Studio в браузере — видно все таблицы
   ```
   Должны появиться таблицы `users`, `products`, `stock_snapshots`, `supplies`,
   `finance_rows`, `product_analytics` и т.д.

**Что можно безопасно повторять:** `npm run db:migrate` (идемпотентно — повторный
запуск ничего не ломает, применяет только новые миграции).

**Что нельзя запускать бездумно:** `npm run db:push` с флагом сброса и любые
ручные `DROP`. Они удаляют данные.

### Если используете Supabase / Neon (managed Postgres)

- Frontend ключи Supabase **не нужны** — WB Shturval не использует Supabase Auth
  (см. [20.9](#209-что-изменено-относительно-этого-тз), решение D-0005). Нужен только
  обычный Postgres.
- В backend пропишите `DATABASE_URL`, который выдаёт Supabase/Neon (вкладка
  Connection string, режим **session/transaction pooler** или прямой).
- Авторизация, redirect URL, RLS настраивать **не нужно** — вход реализован в самом
  backend (email + пароль + JWT в httpOnly-cookie).

## 20.4. Настройка backend

### Способ A (Docker)

Входит в `docker compose up`. Логи:
```bash
docker compose logs -f backend
```
Признак, что backend подключился к базе: в логах есть применённые миграции и строка
`WB Shturval backend listening on :4000`.

### Способ B (вручную)

```bash
cd backend
cp .env.example .env          # заполните DATABASE_URL, JWT_SECRET, ENCRYPTION_KEY
npm install
npm run db:migrate            # применить миграции
npm run dev                   # запуск с авто-перезагрузкой (http://localhost:4000)
```

Проверка здоровья:
```bash
curl http://localhost:4000/health
# {"status":"ok","service":"wb-shturval", ...}
```

Запуск тестов расчётов (важно — это «эталон» формул):
```bash
npm test          # 80 unit-тестов формул снабжения и финансов
```

Деплой backend (раздельный прод): соберите Docker-образ из `backend/Dockerfile`
или запустите на VPS `npm ci && npm run db:migrate && npm start`. Команда
просмотра логов на VPS: `pm2 logs` или `journalctl -u <ваш-сервис>`.

## 20.5. Настройка frontend

### Способ A (Docker)

Входит в `docker compose up`. Сайт: **http://localhost:8080**. nginx внутри
контейнера сам проксирует `/api` на backend, поэтому CORS не задействован.

### Способ B (вручную)

```bash
cd frontend
cp .env.example .env          # для локали оставьте VITE_API_URL пустым
npm install
npm run dev                   # http://localhost:5173 (прокси /api → :4000)
```

Production-сборка:
```bash
npm run build                 # статика в frontend/dist
npm run preview               # локальный предпросмотр сборки
```

Деплой сайта (раздельный прод, напр. Vercel/Netlify):
- соберите `npm run build`, выложите папку `dist`;
- задайте переменную сборки `VITE_API_URL` = адрес вашего backend (напр.
  `https://api.shturval.example.ru`);
- убедитесь, что на backend в `CORS_ORIGIN` добавлен адрес сайта, а `COOKIE_SECURE=true`.

Проверка: откройте сайт, должна появиться форма входа. После входа — дашборд с
предложением подключить WB и запустить синхронизацию.

## 20.6. Настройка синхронизации Wildberries

### Где взять API-ключ

1. Кабинет продавца Wildberries → **Настройки → Доступ к API** (раздел «Новый API»).
2. Создайте токен. **Включите режим «Только на чтение»** (сервис только читает).
3. Отметьте категории:
   - **Статистика** — обязательно (заказы, продажи, остатки, финансовый отчёт);
   - **Контент** — желательно (полный список карточек товаров с фото и категориями);
   - **Продвижение** — желательно (расходы на рекламу для финмодели).
4. Скопируйте токен (это длинная строка-JWT).

### Куда вставить

Сайт → **Настройки → API-ключ Wildberries** → вставьте токен → «Подключить».
Сервис сразу проверит ключ (быстрый тестовый запрос). Если WB ответит 401 — ключ
не примут с понятным сообщением.

### Как запустить первую синхронизацию

Кнопка **«Синхронизировать»** вверху справа. 

Сколько идёт: WB ограничивает статистику до **~1 запроса в минуту на метод**,
поэтому первая полная выгрузка (90 дней заказов/продаж + остатки + финотчёт +
реклама) может занять **от нескольких до 20–40 минут**. Это нормально и идёт в
фоне — можно закрыть вкладку. Статус виден вверху («Идёт синхронизация…») и на
дашборде.

### Частые ошибки и что делать

- **«Wildberries отклонил ключ (401)»** — ключ недействующий, отозван или без
  категории «Статистика». Создайте новый.
- **«Превышение лимитов» / долго** — это норма из-за лимита 1/мин. Подождите.
- **Синхронизация «зависла»** — кнопка **«Сбросить»** (появляется во время выполнения)
  или автоматически сбросится через 30 минут, после чего можно запустить заново.
- **Пустые данные после синка** — проверьте, что у ключа есть категория «Статистика»
  и что в кабинете WB реально есть продажи за последние 90 дней.

## 20.7. Первый запуск после деплоя (чеклист)

1. Откройте сайт (http://localhost:8080 или ваш домен).
2. Зарегистрируйтесь (email + пароль) — первый аккаунт создаётся сразу.
3. Зайдите в **Настройки** → вставьте **API-ключ Wildberries** → «Подключить».
4. (Опционально) задайте параметры снабжения по умолчанию (срок поставки, буфер,
   квант, целевой запас, налог).
5. Нажмите **«Синхронизировать»** и дождитесь завершения.
6. Откройте **Товары** — проверьте, что появились артикулы, остатки, статусы.
7. Заполните **себестоимость**: баннер «Себестоимость не заполнена» → «Импортировать»
   (вставьте таблицу «Артикул ⟶ Себестоимость») или вручную в карточке товара.
8. Откройте **Дашборд** — финансовая сводка, план действий, топ по прибыли.
9. Создайте тестовую **поставку** из карточки товара и проверьте вкладку **В пути**.

## 20.8. Быстрая диагностика

| Симптом | Что проверить в первую очередь |
|---------|-------------------------------|
| **Сайт не открывается** | Запущен ли контейнер `web` (`docker compose ps`); порт 8080 свободен |
| **Frontend не видит backend** | В Docker — работает ли `backend` (`docker compose logs backend`). Раздельно — верный `VITE_API_URL` и `CORS_ORIGIN`, `credentials: include` (уже в коде) |
| **Backend не видит базу** | Верный `DATABASE_URL`; БД поднята и доступна; применены миграции (`npm run db:migrate`) |
| **Авторизация не работает** | Совпадает ли `CORS_ORIGIN` с адресом сайта; под HTTPS должно быть `COOKIE_SECURE=true`; cookie не блокируется браузером |
| **WB-ключ не принимается** | Ключ действующий, режим read-only, есть категория «Статистика»; сообщение об ошибке покажет 401 |
| **Синхронизация падает/зависла** | Раздел [20.6](#206-настройка-синхронизации-wildberries); кнопка «Сбросить»; смотрите статус последнего запуска и поле ошибки |
| **Нет товаров после синка** | Есть ли продажи/остатки за 90 дней в кабинете; статус синка = SUCCESS; категория «Статистика» у ключа |
| **Нет финансовых данных** | У ключа нет доступа к финотчёту, либо WB ещё не сформировал недельный отчёт. На карточке товара будет бейдж «Частичная оценка» |
| **Прибыль не считается** | Не заполнена себестоимость (баннер «Себестоимость не заполнена») — заполните, и пересчёт произойдёт автоматически |
| **Поставки не учитываются** | Статус поставки активный (В пути/Частично); проверьте ожидаемую дату; «Остаток 0, не обнаружена» исключается из рекомендаций намеренно |

Где смотреть логи:
- Docker: `docker compose logs -f backend` (и `web`, `db`).
- Вручную: вывод `npm run dev` в терминале backend.
- Статус и шаги последней синхронизации: дашборд + ответ `GET /api/sync/status`.

## 20.9. Что изменено относительно этого ТЗ

Осознанные отступления (подробности — в `DECISIONS.md`). Формат: изменение →
причина → польза → компромисс.

1. **Единый стек TypeScript (Node/Fastify + React), а не разнесённые языки.**
   *Причина:* в среде подтверждён Node, Python — нет; один язык проще поддерживать.
   *Польза:* общие типы/валидаторы, единый инструментарий, меньше точек отказа.
   *Компромисс:* денежные расчёты на числах float — закрыто округлением на границах
   формул и 80 unit-тестами.

2. **Своя авторизация (email+пароль, JWT в httpOnly-cookie) вместо Supabase Auth.**
   *Причина:* меньше движущихся частей, проще настройка, переносимость.
   *Польза:* не нужно настраивать redirect URL, провайдеров, RLS; БД остаётся любым
   Postgres. *Компромисс:* безопасность входа на нас (закрыто bcrypt + rate-limit).

3. **Backend — долгоживущий сервер с фоновой синхронизацией, а не serverless.**
   *Причина:* выгрузка WB идёт минутами (лимит ~1/мин), serverless-таймауты ненадёжны.
   *Польза:* стабильная фоновая синхронизация со статусом и сбросом зависаний.
   *Компромисс:* при рестарте процесса синк прерывается — закрыто идемпотентностью и
   авто-сбросом зависшего запуска.

4. **Реклама через `GET /adv/v3/fullstats` (а не `POST /adv/v2/fullstats`).**
   *Причина:* WB отключил v2 23.10.2025 (проверено по актуальной документации).
   *Польза:* интеграция работает на текущем API. *Компромисс:* нет.

5. **Скорость продаж считается по «полным» дням (сегодня исключается).**
   *Причина:* незавершённый день занижал бы скорость. *Польза:* стабильная оценка.
   *Компромисс:* продажи «сегодня» не влияют на скорость до конца суток (это
   осознанно; прогноз текущей недели считается отдельно).

6. **День «out-of-stock» исключается только при наличии доказательства (снимок = 0).**
   *Причина:* WB не отдаёт историю остатков, её мы копим сами. *Польза:* честная
   скорость без занижения и без переоценки. *Компромисс:* на первых днях работы
   история остатков копится — точность растёт со временем.

7. **Прибыль не показывается «приблизительно», если нет себестоимости.**
   *Причина:* требование честности данных (спец-раздел 0.4, 11). *Польза:* нет
   ложных прибылей. *Компромисс:* до заполнения себестоимости видна только выручка и
   явный бейдж «Нет данных»/«Частичная оценка».

8. **Расширенные ERP/BI-модули (Cost Engine, закупки/партии, Cash Flow,
   финкалендарь, прогнозы 30/60/90, AI-аналитик) не реализованы в v1.**
   *Причина:* приоритет — стабильное базовое ядро (спец-разделы 0, 0.1).
   *Польза:* надёжный запуск без раздувания. *Компромисс:* часть «витринных» функций
   появится позже; архитектурный фундамент под них уже заложен (слои, история
   себестоимости, версии финданных).

## 20.10. Как сопровождать проект без костылей

Файлы сопровождения лежат в корне:
- `CHANGELOG.md` — что менялось по версиям;
- `BUGFIX_LOG.md` — баги: симптом → **настоящая причина** → фикс → проверка;
- `DECISIONS.md` — архитектурные решения и компромиссы;
- `WILDBERRIES_API_NOTES.md` — актуальный конспект WB API (источник правды интеграции);
- `DESIGN_CODE.md` — дизайн-код (соблюдать на всех экранах).

Правила (обязательны, спец-раздел 0.4):
1. Баг чинится через **поиск причины**, а не маскировку симптома. Результат —
   запись в `BUGFIX_LOG.md` по шаблону.
2. Любой временный обход/fallback — **явно** помечается в коде и заносится в
   технический долг в `DECISIONS.md`. Никаких «тихо вернём пусто/подменим данные».
3. **Одна бизнес-формула — один источник правды.** Все формулы снабжения и
   финансов живут в `backend/src/domain/` (чистые функции). Их используют и
   синхронизация (`recompute`), и карточка товара через общий
   `backend/src/services/economics.ts`. Дублировать формулы запрещено.
4. Слои разделены: загрузка данных (`wb/`, `sync/`) ≠ расчёты (`domain/`) ≠
   отображение (frontend). Не смешивать.
5. После значимого исправления — добавить/обновить проверку: тест в
   `backend/src/domain/*.test.ts`, валидацию или диагностический лог.

Как запускать проверки:
```bash
cd backend
npm test            # unit-тесты формул (эталон расчётов)
npm run typecheck   # проверка типов backend
cd ../frontend
npm run typecheck   # проверка типов frontend
npm run build       # сборка сайта (ловит ошибки бандла)
```

Как понять, что синхронизация и расчёты работают корректно:
- статус последнего `SyncRun` = `SUCCESS`, в `steps` все источники `ok`;
- на дашборде есть выручка/прибыль, статусы товаров расставлены, план действий
  непустой при наличии товаров «к заказу»;
- `npm test` зелёный (80 тестов).

Диагностические состояния (не прячутся от пользователя): ошибка синхронизации,
неполные данные, нет источника, неверный ключ, превышение лимитов WB, пустая
выгрузка, устаревшие данные, неполная финмодель, отсутствующая себестоимость —
видны в интерфейсе (бейджи/баннеры), в статусе синка и в логах.

Что считается временным обходом и требует записи в тех-долг: любое поведение,
которое скрывает ошибку, дублирует формулу, подменяет данные или вводит «магию»
без документации. Если такое появилось — фиксируйте в `DECISIONS.md` с причиной,
риском и планом нормального исправления.

## 20.11. Бесплатный облачный деплой (Supabase + Render + Vercel)

Полностью бесплатный вариант «на первое время», без своего сервера:

- **Supabase** — PostgreSQL (используем только базу, не Supabase Auth).
- **Render** — backend (долгоживущий веб-сервис, бесплатный тариф).
- **Vercel** — сайт (статика) + прокси `/api` на Render.

### Почему именно так (важно про cookie)

Сайт на Vercel и backend на Render — это **разные домены**. Вход хранится в cookie,
а браузеры не отправляют cookie на «чужой» домен из fetch-запросов. Поэтому сайт на
Vercel обращается к **своему же** адресу `/api/...`, а Vercel прозрачно проксирует
это на Render (файл `frontend/vercel.json`). Тогда всё «одного происхождения» —
cookie работает во всех браузерах, CORS не нужен. Это надёжнее, чем cross-site
cookie. (Альтернатива без прокси — в конце раздела.)

### Честно про ограничения бесплатных тарифов

| Сервис | Ограничение | Что это значит | Как смягчить |
|--------|-------------|----------------|--------------|
| **Render free** | сервис «засыпает» после 15 мин без запросов; холодный старт ~30–60 c | первый запрос после простоя — медленный | пока открыта вкладка, она пингует статус синка каждые 4 c и держит сервис активным |
| **Render free** | длинная синхронизация без активных запросов может прерваться при засыпании | если закрыть вкладку во время большого первого синка, он может оборваться | **держите вкладку открытой** во время первой синхронизации; оборванный синк авто-сбрасывается через 30 мин — просто запустите снова |
| **Supabase free** | проект ставится на паузу после ~7 дней без активности; 500 МБ БД | нужно нажать Resume в дашборде | заходите хотя бы раз в неделю; для одного продавца 500 МБ хватает надолго |

Для боевой эксплуатации позже можно поднять Render до самого дешёвого платного
инстанса (без засыпания) — код менять не нужно.

### Шаг 1. База данных — Supabase

1. https://supabase.com → New project. Задайте надёжный **Database password**
   (запомните его).
2. Дождитесь создания, затем **Connect** (кнопка вверху) → вкладка **Session
   pooler** (порт 5432, IPv4-совместимый и совместим с Prisma).
3. Скопируйте строку вида
   `postgresql://postgres.<ref>:<PASSWORD>@aws-0-<region>.pooler.supabase.com:5432/postgres`
   и подставьте вместо `<PASSWORD>` ваш пароль из шага 1. Это будущий `DATABASE_URL`.

   > Берите именно **Session pooler**, а не Transaction pooler — он поддерживает
   > миграции и работу Prisma без дополнительных флагов.

### Шаг 2. Backend — Render

Вариант с Blueprint (в репозитории есть `render.yaml`):

1. https://render.com → **New → Blueprint** → подключите ваш GitHub-репозиторий.
   Render увидит `render.yaml` и предложит создать сервис `wb-shturval-backend`.
2. Заполните переменные, помеченные «требуется значение»:
   - `DATABASE_URL` — строка из Supabase (Шаг 1);
   - `ENCRYPTION_KEY` — сгенерируйте локально:
     `node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"`;
   - `CORS_ORIGIN` — пока можно поставить `*` (уточним на Шаге 4).
   - `JWT_SECRET` Render сгенерирует сам; `COOKIE_SECURE=true`, `COOKIE_SAMESITE=lax`
     уже заданы в blueprint.
3. Создайте сервис и дождитесь сборки. Контейнер при старте сам применит миграции
   (`prisma migrate deploy`) — таблицы создадутся в Supabase автоматически.
4. Проверьте здоровье: откройте `https://<ваш-backend>.onrender.com/health` — должен
   вернуться `{"status":"ok",...}`. Запомните этот адрес backend.

> Без Blueprint: New → **Web Service** → репозиторий → Root Directory `backend`,
> Runtime **Docker**. Те же переменные окружения добавьте вручную, Health Check
> Path = `/health`.

### Шаг 3. Сайт — Vercel

1. Впишите адрес backend в прокси: откройте `frontend/vercel.json`, замените
   `https://YOUR-BACKEND.onrender.com` на ваш адрес с Render, закоммитьте и
   запушьте.
2. https://vercel.com → **Add New → Project** → импортируйте репозиторий.
   - **Root Directory:** `frontend`
   - Framework Preset: **Vite** (определится сам), Build Command `npm run build`,
     Output `dist` — по умолчанию.
   - Environment Variables: **`VITE_API_URL` оставьте пустым** (сайт ходит на
     свой `/api`, который Vercel проксирует на Render).
3. **Deploy.** Получите адрес сайта вида `https://<ваш-проект>.vercel.app`.

### Шаг 4. Связать CORS (необязательно, но правильно)

В Render у сервиса backend задайте `CORS_ORIGIN` = адрес сайта с Vercel
(`https://<ваш-проект>.vercel.app`) и сохраните (сервис перезапустится). При
прокси-схеме CORS почти не задействован, но это аккуратнее.

### Шаг 5. Первый запуск

Дальше — обычный чеклист из [§20.7](#207-первый-запуск-после-деплоя-чеклист):
откройте сайт Vercel → регистрация → Настройки → WB-ключ → «Синхронизировать»
(**держите вкладку открытой**, пока идёт первый большой синк).

### Альтернатива без Vercel-прокси (cross-site)

Если не хотите прокси: на Vercel задайте `VITE_API_URL` = адрес backend на Render,
а на Render поставьте `COOKIE_SAMESITE=none` (и `COOKIE_SECURE=true`, и `CORS_ORIGIN`
= точный адрес сайта Vercel). Тогда сайт обращается к Render напрямую. Минус:
cross-site cookie могут блокироваться строгими браузерами (Safari/ITP, будущие
ограничения Chrome), поэтому прокси-схема выше надёжнее.
