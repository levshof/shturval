# Wildberries API — рабочий конспект (WB Shturval)

> Источник правды для интеграции. Перед изменением кода интеграции сверяйтесь с
> этим файлом и с официальной документацией. Если живая документация Wildberries
> расходится с этим конспектом — **приоритет у живой документации**, а конспект
> нужно обновить с указанием даты и причины.

**Дата сбора:** 2026-06-30
**Кем собрано:** автоматический разбор официальных и сообществом подтверждённых
источников (см. раздел «Источники»).

## 0. Статус проверки

Официальный портал `dev.wildberries.ru` и `openapi.wb.ru` отдают контент только
после JS-проверки и блокируют простые HTTP-запросы (HTTP 498 / ECONNREFUSED),
поэтому часть деталей собрана из официальных страниц через поисковые сниппеты и
из подтверждённых сообществом источников. Каждый блок помечен статусом:

- `✅ verified` — подтверждено несколькими независимыми источниками.
- `🟡 check-live` — общеизвестно, но перед продакшеном свериться с живой докой.

Все эндпоинты, которые реально вызывает код, инкапсулированы в одном клиенте
(`backend/src/wb/client.ts`). Если WB поменяет адрес/версию — правка в одном месте.

## 1. Авторизация и типы токенов `✅ verified`

- Токен Wildberries — это **JWT**, который продавец создаёт в личном кабинете:
  `Профиль → Настройки → Доступ к API` (раздел может называться
  «Настройки → Доступ к новому API»).
- При создании токена выбираются **категории доступа**. Один токен может содержать
  **несколько категорий**. Есть отдельный флаг **«Только на чтение»** (read-only) —
  с ним нельзя ничего изменять, только получать данные. **Для WB Shturval нужен
  именно read-only токен**, потому что сервис только читает статистику.
- Категории, которые нужны WB Shturval:
  - **Статистика** — заказы, продажи, остатки, финансовый отчёт по реализации.
  - **Контент** — список карточек товаров (артикулы, nmID, категории).
  - **Продвижение** — расходы на рекламу (опционально, для финмодели).
  - (опц.) **Аналитика** — воронка продаж; в базовом ядре **не используется**.
- Передаётся в HTTP-заголовке:

  ```http
  Authorization: <JWT-токен целиком>
  ```

  Историческая заметка: исторически токен передавался без префикса `Bearer`.
  Современный API принимает «голый» токен в `Authorization`. Наш клиент шлёт токен
  как есть, без `Bearer ` (если WB вернёт 401 на валидный ключ — это первое, что
  нужно проверить, см. BUGFIX-процесс).
- Срок жизни токена — до 180 дней; при компрометации токен можно отозвать в ЛК.

> Замечание по безопасности: токен даёт доступ к данным магазина. Храним его на
> бэкенде в зашифрованном виде (AES-256-GCM), **никогда** не отдаём на фронтенд,
> в БД лежит шифротекст. См. `DECISIONS.md` → «Хранение WB-ключа».

## 3. Статистика — заказы `✅ verified`

```
GET https://statistics-api.wildberries.ru/api/v1/supplier/orders?dateFrom={RFC3339}&flag=0
```

- **Лимит запросов:** не более **1 запроса в минуту** на аккаунт продавца.
- **Глубина данных:** гарантируется хранение **не более 90 дней** от даты заказа.
  Данные обновляются примерно раз в 30 минут.
- **`flag`:**
  - `flag=0` (или не указан) — возвращаются строки, у которых `lastChangeDate >= dateFrom`.
    Это режим инкрементальной выгрузки «всё, что изменилось с момента X».
  - `flag=1` — возвращаются все заказы с **датой заказа == dateFrom** (время игнорируется).
- **Пагинация:** на один ответ условный лимит **80 000 строк**. Если пришло
  «много» строк — берём **полное значение `lastChangeDate` последней строки** и
  подставляем его в `dateFrom` следующего запроса. Повторяем, пока ответ не
  «исчерпается» (см. защиту от зацикливания ниже).
- **Идентификатор:** заказ идентифицируется полем **`srid`**. 1 строка = 1 заказ = 1 товар.
- **Отмены:** поле `isCancel` (bool) и `cancelDate`. **Отменённые заказы не должны
  искажать спрос** (фильтруем при расчёте скорости продаж).
- Ключевые поля ответа:

  | Поле               | Смысл                                            |
  |--------------------|--------------------------------------------------|
  | `date`             | дата-время заказа (RFC3339)                      |
  | `lastChangeDate`   | дата-время последнего изменения (курсор)         |
  | `supplierArticle`  | артикул продавца (vendorCode)                    |
  | `nmId`             | артикул WB (nmID)                                |
  | `barcode`          | штрихкод (баркод размера)                        |
  | `techSize`         | размер                                           |
  | `category`,`subject`,`brand` | категория / предмет / бренд            |
  | `warehouseName`    | склад                                            |
  | `totalPrice`       | цена до скидки                                   |
  | `discountPercent`  | скидка, %                                        |
  | `spp`              | скидка постоянного покупателя, %                 |
  | `finishedPrice`    | цена с учётом всех скидок (фактическая)          |
  | `priceWithDisc`    | цена со скидкой продавца                          |
  | `isCancel`         | признак отмены                                   |
  | `srid`             | идентификатор заказа                             |

### Защита от бесконечной пагинации `✅ verified` (наше правило)

Курсор `lastChangeDate` монотонно растёт, но на границах может «застрять» (две
строки с одинаковым `lastChangeDate`). Правила клиента:
1. Если новый ответ пустой — стоп.
2. Если `lastChangeDate` последней строки == курсору предыдущего запроса **и**
   количество строк == размеру страницы — сдвигаем курсор на +1 секунду, чтобы не
   зациклиться (документируем как осознанный обход, см. BUGFIX_LOG при появлении).
3. Жёсткий предохранитель: максимум N страниц на одну выгрузку (конфиг).

## 4. Статистика — продажи `✅ verified`

```
GET https://statistics-api.wildberries.ru/api/v1/supplier/sales?dateFrom={RFC3339}&flag=0
```

- Лимит, `flag`, пагинация, глубина 90 дней — **как у заказов**.
- **Различение продаж и возвратов:** поле **`saleID`**:
  - начинается с **`S`** — продажа;
  - начинается с **`R`** — возврат;
  - (встречается `D` — для некоторых корректировок).
  **Возвраты не считаются обычными продажами** (исключаем при расчёте скорости и
  выручки; учитываем отдельно).
- Доп. поле **`forPay`** — сумма к перечислению продавцу по строке (грубая оценка
  выплаты до финансового отчёта).
- Поля цены те же: `finishedPrice`, `priceWithDisc`, `totalPrice`, `discountPercent`.
  Используются в fallback-цепочке цены (см. `DECISIONS.md` и `finance.ts`).

## 5. Статистика — остатки `✅ verified`

```
GET https://statistics-api.wildberries.ru/api/v1/supplier/stocks?dateFrom={RFC3339}
```

- Возвращает **текущий срез остатков** (не историю). Чтобы получить «всё»,
  передаём раннюю дату (например `2019-06-20`).
- **Истории остатков в API нет** — её строим сами: на каждой синхронизации
  сохраняем дневной снимок (`stock_snapshots`). Это нужно для «доступных дней» и
  честной скорости продаж (спец-раздел 6.1).
- Ключевые поля:

  | Поле             | Смысл                                             |
  |------------------|---------------------------------------------------|
  | `nmId`,`supplierArticle`,`barcode` | идентификаторы                  |
  | `warehouseName`  | склад                                             |
  | `quantity`       | доступно к продаже на складе                       |
  | `inWayToClient`  | едет к клиенту                                     |
  | `inWayFromClient`| едет от клиента (возвраты в пути)                  |
  | `quantityFull`   | полное количество (с учётом в пути)               |
  | `Price`,`Discount` | цена/скидка                                     |

- Текущий остаток товара = сумма `quantity` по всем складам для nmId
  (политику фиксируем в `DECISIONS.md`).

> Замечание (изменение WB от 09.02): для управления остатками WB отказывается от
> привязки к баркодам в пользу `chrtId` (ID размера). Для **чтения** среза остатков
> метод `supplier/stocks` остаётся рабочим; агрегируем по `nmId`. `🟡 check-live`.

## 6. Финансовый отчёт по реализации `✅ verified`

```
GET https://statistics-api.wildberries.ru/api/v5/supplier/reportDetailByPeriod
    ?dateFrom={YYYY-MM-DD}&dateTo={YYYY-MM-DD}&rrdid=0&limit=100000
```

- **Главный источник денег.** WB формирует отчёты **по неделям**
  (понедельник–воскресенье). Это основа «проверенных недель» (спец-раздел 10).
- **Лимит:** `limit` ≤ **100000**. Запросов — не чаще **1/мин**.
- **Пагинация по `rrdid`:** берём `max(rrd_id)` из последнего ответа и передаём как
  `rrdid` в следующий запрос. Пустой ответ → конец.
  ⚠️ **Важный нюанс:** на границах недель/отчётов `rrd_id` не строго монотонен,
  поэтому нельзя предполагать сортировку. Наше правило: тянем все строки, на нашей
  стороне дедупим по `rrd_id` и агрегируем по неделе/nmId.
- Ключевые поля (денежные):

  | Поле                    | Смысл                                                    |
  |-------------------------|----------------------------------------------------------|
  | `realizationreport_id`  | id отчёта (недели)                                        |
  | `rrd_id`                | id строки (курсор пагинации)                              |
  | `date_from`,`date_to`   | период отчёта (неделя)                                    |
  | `nm_id`,`sa_name`,`barcode` | идентификаторы товара                                |
  | `doc_type_name`         | «Продажа» / «Возврат» и т.п.                              |
  | `quantity`              | количество                                               |
  | `retail_price`          | розничная цена                                           |
  | `retail_amount`         | сумма продажи (розница)                                  |
  | `ppvz_for_pay`          | **к перечислению продавцу** по строке (ключевое)         |
  | `ppvz_sales_commission` | комиссия WB за продажу                                   |
  | `delivery_rub`          | логистика, руб.                                          |
  | `storage_fee`           | хранение, руб.                                           |
  | `penalty`               | штрафы                                                   |
  | `deduction`             | удержания                                               |
  | `acceptance`            | платная приёмка                                          |
  | `return_amount`         | сумма возвратов                                          |
  | `rebill_logistic_cost`  | корректировки логистики                                  |

- **Стратегия расчёта прибыли:** при наличии отчёта используем
  `seller_payout = Σ ppvz_for_pay` как «выплату продавцу», что надёжнее, чем
  собирать расходы по частям. Полная формула — в `finance.ts` (единый источник).
  Расходы WB как сумма частей — резервный путь, когда нет надёжной выплаты.

## 7. Контент — список карточек товаров `🟡 check-live`

```
POST https://content-api.wildberries.ru/content/v2/get/cards/list
Body: { "settings": { "cursor": { "limit": 100 }, "filter": { "withPhoto": -1 } } }
```

- Возвращает мастер-список товаров: `nmID`, `vendorCode` (= supplierArticle),
  `brand`, `title`, `subjectName`/`subjectID` (категория/предмет), размеры со `skus`
  (баркоды), фото, габариты.
- **Курсорная пагинация:** в ответе `cursor` с `updatedAt` и `nmID`; их кладём в
  `filter`-курсор следующего запроса; конец — когда вернулось меньше `limit`.
- Используем как **источник справочника товаров** (а не статистику), чтобы знать
  все артикулы, даже без продаж/остатков.

> Если контент-токена нет — деградируем мягко: справочник товаров собираем из
> заказов/продаж/остатков (там есть `nmId`+`supplierArticle`+`category`). Тогда не
> будет товаров без активности. Это **явный**, документированный режим, не «тихий
> fallback» (помечаем источник товара).

## 8. Продвижение — расходы на рекламу `✅ verified (важное изменение!)`

- **Список кампаний:** `GET https://advert-api.wildberries.ru/adv/v1/promotion/count`
- **Статистика (НОВЫЙ метод):**

  ```
  GET https://advert-api.wildberries.ru/adv/v3/fullstats
      ?ids={id1,id2,...}&beginDate=YYYY-MM-DD&endDate=YYYY-MM-DD
  ```

  - ⚠️ **Старый `POST /adv/v2/fullstats` отключён 23.10.2025.** Нужно использовать
    **`GET /adv/v3/fullstats`** с параметрами `ids` (через запятую), `beginDate`,
    `endDate` в формате `YYYY-MM-DD`.
  - В v3 добавлено поле **`canceled`** (технические отмены: товара нет в наличии,
    ошибки доставки) — это **не** отказ покупателя.
  - Ответ содержит разбивку по дням и по `nmId` с метриками: показы (`views`),
    клики (`clicks`), затраты (`sum`), заказы (`orders`).
  - **`🟡 check-live`:** по опыту продавцов на форуме WB API (`dev.wildberries.ru/forum/1441`,
    тема «CRITICAL BUG: /adv/v3/fullstats возвращает 0 в sum/views/clicks») метод
    может отдавать реальный дневной итог по кампании (`views`/`clicks`/`sum` на
    уровне `days[]`), но нулевую или неполную разбивку по `nm[]` — особенно на
    автокампаниях. **Подтверждено на живом ключе (2026-07-02):** метод отдаёт
    структуру кампаний, но `sum`/`views`/`clicks` = 0 на всех уровнях (это не
    «нет трат» — трат много, видно в кабинете). Поэтому `days[].sum` из fullstats
    **нельзя** использовать как источник расхода — берём его из `/adv/v1/upd`
    (ниже), а `nm[]` из fullstats оставляем только для разбивки по товарам.
- **Реальный расход (источник правды) — `GET https://advert-api.wildberries.ru/adv/v1/upd?from=YYYY-MM-DD&to=YYYY-MM-DD`** `✅ verified (2026-07-02)`
  - «История затрат» (списания по кампаниям). **Не** затронута багом fullstats.
    Лимит ~1 запрос/сек; один запрос покрывает всё окно, фильтруем по дате у себя.
  - Ответ — массив списаний: `updSum` (сумма, ₽), `advertId` (кампания),
    `updTime` (RFC3339 `+03:00`, может быть `null` у ещё не списанных строк),
    `campName`, `advertType`, `paymentType`, `advertStatus`. Дневной расход
    кампании = Σ `updSum` по `advertId` + день (`updTime` в MSK).
- **Привязка к товару:** реальный расход берём из `/adv/v1/upd` (по кампаниям и
  дням), разбивку по `nmId` — из `nm[]` fullstats. Точный per-nm расход (когда
  fullstats не занулён) идёт как `PRECISE`; неразнесённый остаток кампании
  распределяем по её товарам пропорционально их доле точного расхода, а если
  точного расхода нет вовсе — поровну между товарами кампании
  (`AdSource.ALLOCATED`, спец-раздел 9.4, BUG-0004/0006). Если у кампании вообще
  нет ни одной строки с товаром за весь период — остаток не выдумывается, а
  попадает в диагностику шага синка «реклама» (unattributed).

## 9. Форматы дат `✅ verified`

- Статистика (orders/sales/stocks): `dateFrom` в **RFC3339**
  (`2024-01-01T00:00:00Z` или `2024-01-01T00:00:00`). Допускается короткий
  `YYYY-MM-DD`. Курсор `lastChangeDate` подставляем **в точности как пришёл**.
- `reportDetailByPeriod`: `dateFrom`/`dateTo` в **`YYYY-MM-DD`**.
- Реклама v3: `beginDate`/`endDate` в **`YYYY-MM-DD`**.
- Часовой пояс данных WB — **Москва (MSK, UTC+3)**. Все «дни» в продуктовой логике
  считаем в MSK, чтобы «сегодня» совпадало с днём продавца (фиксируем в DECISIONS).

## 10. Лимиты и обработка ошибок `✅ verified / 🟡 check-live`

- Статистические эндпоинты: **1 запрос/мин** на каждый метод. Клиент обязан
  соблюдать интервал (rate limiter с очередью по хосту/методу).
- Коды:
  - `401` — невалидный/просроченный токен или нет нужной категории.
  - `429` — превышен лимит; уважать заголовок `Retry-After` (или `X-Ratelimit-Retry`).
  - `400` — неверные параметры (даты, лимит).
  - `5xx` — временная ошибка WB → ретрай с экспоненциальной паузой.
- Наш клиент:
  - очередь + интервал на хост;
  - ретраи с бэкоффом на `429`/`5xx` (ограниченное число);
  - на `401` — **не ретраим**, помечаем ключ как невалидный и показываем
    пользователю явный статус (спец-раздел 0.4, диагностические состояния).

## 11. Что это значит для синхронизации WB Shturval

- Полная синхронизация — это серия медленных (≥1 мин/метод) последовательных
  выгрузок. Это **долгая фоновая задача**, а не запрос «туда-обратно». Поэтому
  бэкенд — **долгоживущий сервер** с фоновым воркером и таблицей статусов
  (`sync_runs`), а не serverless-функция (см. `DECISIONS.md`).
- Первый запуск тянет ~90 дней заказов/продаж и несколько недель финотчёта —
  это десятки минут из-за лимита 1/мин. Пользователю показываем прогресс и шаги.
- **Защита данных (спец-раздел 12):** если ключевой источник пришёл подозрительно
  неполным (резко меньше заказов/артикулов, оборвалась пагинация, пустые остатки) —
  **не публикуем** плохую выгрузку, а показываем ошибку синхронизации и оставляем
  прошлый хороший срез. Реализация — `sync/guards.ts`.

## 12. Открытые вопросы / проверить на живой доке `🟡`

1. Точные актуальные численные лимиты (RPS/в минуту) на `content` и `advert` —
   свериться в `dev.wildberries.ru/openapi`.
2. Полная схема ответа `adv/v3/fullstats` (вложенность `days[] → apps[] → nm[]`),
   в частности — **действительно ли `days[]` несёт собственные `sum`/`views`/`clicks`**
   отдельно от разбивки по `nm[]` (используется для BUG-0004, см. раздел 8).
3. Нужен ли префикс `Bearer` для отдельных новых сервисов.
4. Изменения по остаткам (переход на `chrtId`) — влияет ли на чтение `supplier/stocks`.
