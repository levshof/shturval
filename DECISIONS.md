# DECISIONS.md — WB Shturval

Важные архитектурные и продуктовые решения: что выбрали, почему, какие альтернативы
отклонили, какие компромиссы. Формат записи фиксированный. Новые решения — сверху
по дате.

---

## D-0001 — Технологический стек: единый TypeScript

**Дата:** 2026-06-30
**Решение:** Backend — Node.js + TypeScript (Fastify). Frontend — React + TypeScript
(Vite). БД — PostgreSQL через Prisma. Валидация — Zod. Тесты — Vitest.
**Почему:**
- Один язык на весь проект → проще поддержка для одного владельца, общие типы и
  валидаторы (Zod) между фронтом и бэком, единый инструментарий.
- Node 24 и npm 11 уже установлены и проверены в среде; Python — нет.
- Prisma даёт декларативную схему + воспроизводимые миграции (важно для надёжной,
  повторяемой настройки БД из `SETUP_GUIDE.md`).
**Альтернативы отклонены:**
- Python/FastAPI — отличен для расчётов, но добавляет второй язык/тулчейн и не
  подтверждён в среде.
- Next.js (full-stack) — проще деплой, но serverless плохо подходит для долгой (минуты)
  фоновой синхронизации WB (лимит 1 запрос/мин). Нужен долгоживущий сервер.
**Компромисс:** TS-расчёты требуют аккуратной работы с числами (float). Митигируем
округлением на границах формул и тестами на формулы.

## D-0002 — Бэкенд как долгоживущий сервер с фоновым воркером (не serverless)

**Дата:** 2026-06-30
**Решение:** Один Node-процесс держит HTTP API и выполняет синхронизацию как
фоновую задачу внутри процесса; статус и шаги пишутся в таблицу `sync_runs`.
**Почему:** WB Statistics — 1 запрос/мин на метод; полная выгрузка занимает десятки
минут. Это фоновый процесс с прогрессом, а не запрос-ответ. Serverless-таймауты тут
ненадёжны.
**Альтернативы отклонены:** очередь + отдельный воркер (BullMQ/Redis) — мощнее, но
лишняя инфраструктура (Redis) на старте. Заложили интерфейс `JobRunner`, чтобы
позже вынести в очередь без переписывания логики.
**Компромисс:** при рестарте процесса во время синка задача прервётся. Митигируем:
синк идемпотентный (upsert по натуральным ключам), статус «running» с временем старта
и кнопкой «сбросить зависшую синхронизацию» (спец-раздел 5.2).

## D-0003 — Хранение WB-ключа: шифрование на бэкенде, никогда на фронт

**Дата:** 2026-06-30
**Решение:** WB API-ключ шифруется AES-256-GCM ключом из `ENCRYPTION_KEY` (env) и
хранится как шифротекст в БД. На фронт отдаётся только факт «ключ подключён» и маска.
**Почему:** ключ — доступ к данным магазина. В коде/БД в открытом виде — недопустимо
(спец-раздел 20.2: секреты не хардкодить).
**Компромисс:** при утере `ENCRYPTION_KEY` ключи не расшифровать — их нужно ввести
заново. Это приемлемо и безопаснее хранения в открытую. `ENCRYPTION_KEY` — в секретах
хостинга.

## D-0004 — Единый источник правды для формул (анти-дублирование)

**Дата:** 2026-06-30
**Решение:** Вся бизнес-математика — в `backend/src/domain/` (модули `supply.ts`,
`finance.ts`, `forecast.ts`) как **чистые функции** без обращения к БД/сети/UI.
Каждая формула определена **ровно один раз** и покрыта unit-тестами (Vitest).
**Почему:** спец-раздел 0.4 запрещает дублирование формул и смешивание загрузки
данных, расчётов и отображения. Чистые функции легко тестировать и переиспользовать.
**Как применяем:** сервисы загружают данные → передают в domain-функции → отдают
результат в API. Фронт **не** пересчитывает бизнес-логику, только форматирует.
**Компромисс:** часть расчётов могла бы жить в SQL (быстрее на больших объёмах). На
старте приоритет — читаемость и тестируемость; при росте объёмов вынесем «тяжёлые»
агрегации в SQL-вьюхи, сохранив domain как эталон.

## D-0005 — Авторизация: собственный JWT (email + пароль), не Supabase Auth

**Дата:** 2026-06-30
**Решение:** Регистрация/вход по email+паролю; пароль — bcrypt; сессия — JWT
(access-токен) в httpOnly-cookie. Один сервис, без внешнего провайдера.
**Почему:** меньше движущихся частей и проще `SETUP_GUIDE` (не нужно настраивать
redirect URL, провайдеров, RLS). Продукт самодостаточен и переносим между хостингами.
**Альтернативы отклонены:** Supabase Auth — упоминался в ТЗ как пример, но привязывает
к конкретному провайдеру и усложняет настройку. БД при этом всё равно обычный Postgres
(можно Supabase/Neon/локально) — выбор хостинга БД остаётся свободным.
**Компромисс:** мы сами отвечаем за безопасность auth (хеши, ротация секрета,
блокировка перебора). Митигируем проверенными библиотеками и базовым rate-limit на
вход.

## D-0006 — Постоянное хранилище: PostgreSQL + дневные снимки остатков

**Дата:** 2026-06-30
**Решение:** PostgreSQL. Остатки храним как **дневные снимки** (`stock_snapshots`),
заказы/продажи — построчно (по `srid`/`saleID`), финотчёт — построчно (по `rrd_id`)
с агрегатами по неделе/nmId.
**Почему:** WB не отдаёт историю остатков — её строим сами (нужно для «доступных дней»
и честной скорости продаж, спец-раздел 6.1). Postgres надёжен, есть JSON, транзакции,
индексы.
**Компромисс:** снимки растут со временем. Митигируем индексами и (позже) политикой
ретеншена/агрегации старых снимков. На объёмах одного продавца это незначительно.

## D-0007 — Идемпотентная и «защищённая» синхронизация

**Дата:** 2026-06-30
**Решение:** Все записи синхронизации — upsert по натуральным ключам WB. Перед
публикацией новой выгрузки прогоняем «guards»: резкое падение числа заказов/артикулов,
оборванная пагинация, пустые остатки → выгрузка помечается подозрительной и **не
заменяет** прошлый хороший срез; пользователь видит ошибку синхронизации.
**Почему:** спец-раздел 12 — лучше показать ошибку, чем тихо опубликовать плохие данные.
**Компромисс:** ложные срабатывания guard на резком (но реальном) изменении бизнеса.
Митигируем: пороги в конфиге + возможность «принудительно принять» с явным
подтверждением (документируется при добавлении).

## D-0008 — Объём первой версии: надёжное базовое ядро, фундамент под ERP/BI

**Дата:** 2026-06-30
**Решение:** Реализуем базовое ядро (спец-раздел 0.1) полностью и надёжно. Расширенные
модули (Cost Engine, закупки/партии, Cash Flow, финкалендарь, прогнозы 30/60/90,
AI-аналитик) — **не** реализуем в v1, но закладываем фундамент: разделение слоёв,
история себестоимости, версионируемые финданные, нейтральные доменные интерфейсы.
**Почему:** приоритет — стабильный базовый продукт без раздувания (спец-разделы 0, 0.1).
**Компромисс:** часть «витринных» функций появится позже. Это осознанно.

## D-0009 — Прогноз текущей недели как отдельный слой с уровнем уверенности

**Дата:** 2026-06-30
**Решение:** Финмодель строго разделяет «проверенные недели» (из `reportDetailByPeriod`)
и «прогноз текущей недели». Прогноз помечается уверенностью (высокая/средняя/низкая) и
визуально отличается на графиках (пунктир/штриховка, см. DESIGN_CODE §10).
**Почему:** спец-разделы 9–11 — пользователь должен видеть качество данных и не путать
факт с прогнозом.
**Компромисс:** больше состояний в UI и расчётах. Оправдано требованием честности.

## D-0010 — Деплой на бесплатный стек: same-origin через Vercel-прокси

**Дата:** 2026-07-01
**Решение:** Поддержан бесплатный облачный вариант Supabase (Postgres) + Render
(backend) + Vercel (сайт). Сайт обращается к своему `/api`, а Vercel проксирует на
Render (`frontend/vercel.json`) → всё «одного происхождения». Cookie сессии остаётся
`SameSite=Lax`. Добавлен конфиг-ключ `COOKIE_SAMESITE` для cross-site-альтернативы.
**Почему:** Vercel и Render — разные домены; cross-site cookie ненадёжны
(Safari/ITP, ограничения Chrome). Прокси делает запросы first-party во всех браузерах,
без CORS. Render — долгоживущий сервис, поэтому фоновый синк работает (в отличие от
serverless), а Supabase Session pooler совместим с Prisma и миграциями.
**Альтернативы отклонены:** прямой cross-site (`VITE_API_URL`=backend +
`SameSite=None`) — оставлен как задокументированная опция, но не по умолчанию.
**Компромисс:** бесплатные тарифы засыпают/ставятся на паузу (Render 15 мин,
Supabase ~7 дней); длинный синк нужно «досмотреть» с открытой вкладкой. Оборванный
синк авто-сбрасывается через 30 мин. Для прода — дешёвый платный инстанс Render без
изменения кода.
